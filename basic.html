قسم BASIC — الأساس الواضح قبل Oracle SQL

هذا القسم مبني بطريقة تعليمية مفصلة. كل فكرة تبدأ بتعريف بسيط، ثم مثال من الحياة، ثم رسم، ثم تطبيق، ثم أخطاء شائعة. الهدف أنك تفهم وليس فقط تحفظ.

1) Data & Database — البيانات وقاعدة البيانات

Data هي حقائق خام مثل: اسم طالب، رقم هوية، تاريخ، سعر، أو درجة اختبار. لوحدها قد لا تعطي معنى كامل.

Information هي بيانات تم ترتيبها ومعالجتها لتساعدك في اتخاذ قرار. مثال: درجة الطالب 92 تصبح معلومة عندما نقول: الطالب ناجح بتقدير ممتاز.

لماذا نحتاج Database؟

لأن الملفات العادية و Excel تصبح صعبة عند كثرة البيانات، تعدد المستخدمين، الحاجة للبحث السريع، حماية البيانات، ومنع التكرار والأخطاء.

مثال حقيقي

جامعة لديها طلاب، مواد، درجات، مدرسين، شعب. لو خزنت كل شيء في ملف واحد ستتكرر البيانات ويصعب استخراج التقارير. قاعدة البيانات تقسمها لجداول مرتبطة.

خطأ شائع: الاعتقاد أن قاعدة البيانات مجرد جدول Excel. الفرق أن قاعدة البيانات تدعم العلاقات، القيود، الصلاحيات، الاستعلامات، والتعامل مع عدد كبير من المستخدمين.

2) Database Types — أنواع قواعد البيانات

Relational Database

تخزن البيانات في جداول مترابطة. ممتازة للأنظمة التي تحتاج علاقات واضحة مثل الجامعات والبنوك والفواتير. أمثلة: Oracle, MySQL, SQL Server, PostgreSQL.

Document Database

تخزن البيانات كوثائق JSON مرنة. مناسبة للتطبيقات التي تتغير بياناتها كثيرًا مثل بعض تطبيقات الويب. مثال: MongoDB, Firebase Firestore.

Key-Value

تخزن البيانات بشكل مفتاح وقيمة. سريعة جدًا للكاش والجلسات. مثال: Redis.

Graph Database

مناسبة للعلاقات المعقدة مثل الشبكات الاجتماعية والتوصيات. مثال: Neo4j.

3) SQL vs NoSQL — الفرق بين SQL و NoSQL

SQL هو أسلوب للتعامل مع قواعد البيانات العلائقية. يعتمد على جداول وصفوف وأعمدة وعلاقات. NoSQL مصطلح يشمل قواعد لا تعتمد دائمًا على الجداول التقليدية.

النقطةSQLNoSQL
الشكلTablesDocuments / Key-Value / Graph
العلاقاتقوية وواضحةمرنة حسب النوع

مناسب لـبنوك، جامعات، فواتيرتطبيقات مرنة، كاش، بيانات ضخمة
قاعدة سهلة: إذا كان النظام يعتمد على علاقات وتقارير دقيقة، ابدأ بـ SQL. إذا كانت البيانات متغيرة جدًا أو تحتاج توسعًا أفقيًا مرنًا، قد يناسبك NoSQL.

4) SRS — Software Requirements Specification

وثيقة SRS هي مستند يشرح ماذا يجب أن يفعل النظام قبل التصميم والبرمجة. هي الجسر بين فكرة العميل وفريق التطوير.

مكونات SRS الأساسية

أشياء يجب أن يفعلها النظام. مثال: يستطيع الطالب تسجيل الدخول، يستطيع الموظف إضافة فاتورة، يستطيع المدير طباعة تقرير.

صفات النظام: السرعة، الأمان، سهولة الاستخدام، الاعتمادية، قابلية التوسع. مثال: يجب أن تظهر نتيجة البحث خلال أقل من ثانيتين.

من يستخدم النظام؟ وما المهام التي يقوم بها؟ مثال: Student, Admin, Teacher.

مثال SRS لنظام مكتبة

النظام يسمح للموظف بإضافة كتاب، وللعضو باستعارة كتاب، ويجب ألا تتم الاستعارة إذا كانت الكمية غير متوفرة. يجب أن يحفظ النظام تاريخ الاستعارة وتاريخ الإرجاع.

5) ERD — Entity Relationship Diagram

ERD هو مخطط يوضح الكيانات في النظام والعلاقات بينها قبل تحويلها إلى جداول. هو مثل خريطة النظام. إذا كان ERD صحيحًا، غالبًا سيكون تصميم قاعدة البيانات صحيحًا.

قبل ERD: كيف نفكر؟

نقرأ المتطلبات ثم نسأل: ما الأشياء المهمة التي نحتاج تخزينها؟ هذه الأشياء تصبح Entities. ثم نسأل: ما صفات كل شيء؟ هذه Attributes. ثم نسأل: كيف ترتبط هذه الأشياء؟ هذه Relationships.

Entity — الكيان

شيء نحتاج حفظ بيانات عنه. مثل Student, Employee, Product, Invoice. غالبًا يتحول إلى جدول.

Attribute — الصفة

معلومة تصف الكيان. مثل name, salary, price. غالبًا تتحول إلى عمود داخل الجدول.

Primary Key

عمود يميز كل صف عن غيره. مثل employee_id. لا يتكرر ولا يكون فارغًا.

Foreign Key

عمود يربط جدولًا بجدول آخر. مثل department_id داخل EMPLOYEE يشير إلى DEPARTMENT.

أنواع العلاقات

One to One
PERSON1 — 1PASSPORT

الشخص لديه جواز واحد، والجواز لشخص واحد.

One to Many
DEPARTMENT1 — NEMPLOYEE

القسم يحتوي عدة موظفين، والموظف في قسم واحد.

Many to Many
STUDENTN — MCOURSE

الطالب يأخذ عدة مواد، والمادة يأخذها عدة طلاب. نحلها بجدول وسيط ENROLLMENT.

6) Schema — السكيما

Schema هي مساحة تنظيمية داخل قاعدة البيانات تحتوي على جداول و Views و Sequences و Objects أخرى. في Oracle غالبًا ترتبط السكيما بالمستخدم. مثال HR Schema تحتوي على EMPLOYEES و DEPARTMENTS و JOBS.

7) القواعد السبعة لبناء جدول صحيح

قبل كتابة CREATE TABLE، لازم تفهم كيف تصمم الجدول. الجدول الجيد واضح، غير مكرر، سهل الربط، وسهل الاستعلام.

1

اسم واضح للجدول

استخدم اسم يدل على الكيان مثل STUDENT أو EMPLOYEE. تجنب أسماء غامضة مثل T1 أو DATA.

2

Primary Key لكل جدول

كل جدول يحتاج عمود يميز كل صف. مثال student_id.

3

أعمدة واضحة

اكتب first_name بدل fn، وhire_date بدل hd.

4

اختر Data Type صحيح

الأسماء VARCHAR2، الرواتب NUMBER، التواريخ DATE.

5

لا تكرر البيانات

إذا كان department_name يتكرر مع كل موظف، افصل DEPARTMENT في جدول مستقل.

6

استخدم Foreign Keys

لربط الجداول وحماية العلاقات من الأخطاء.

7

استخدم Naming Convention

مثال: employee_id, department_id, hire_date. نفس النمط يجعل النظام مفهومًا.

-- مثال تصميم منطقي قبل إنشاء الجدول
TABLE: EMPLOYEE
PK: employee_id
Columns: first_name, last_name, salary, hire_date
FK: department_id → DEPARTMENT.department_id

8) HR Schema — نموذج Oracle التدريبي

HR Schema تستخدم في تدريب Oracle لأنها تحتوي على جداول مترابطة تمثل نظام موارد بشرية.

EMPLOYEES

employee_id, first_name, last_name, email, hire_date, job_id, salary, department_id

DEPARTMENTS

department_id, department_name, manager_id, location_id

JOBS

job_id, job_title, min_salary, max_salary

LOCATIONS

location_id, street_address, city, country_id

عندما تكتب FROM employees فأنت تطلب البيانات من جدول EMPLOYEES داخل HR Schema.

9) Basic SELECT — أول استعلام

SELECT يستخدم لاسترجاع البيانات من جدول. هو أول أمر تتعلمه في SQL.

SELECT *
FROM employees;

شرح كل جزء

  • SELECT: اختر الأعمدة التي تريد عرضها.
  • *: يعني عرض كل الأعمدة.
  • FROM: من أي جدول؟
  • employees: اسم الجدول.
  • ;: نهاية الأمر في أدوات كثيرة مثل SQL Developer.
SELECT first_name, salary
FROM employees;
خطأ شائع: استخدام SELECT * دائمًا. في المشاريع الحقيقية الأفضل تحديد الأعمدة المطلوبة لتقليل البيانات وتحسين الوضوح.

10) WHERE & ORDER BY — التصفية والترتيب

WHERE يستخدم لتصفية الصفوف حسب شرط. ORDER BY يستخدم لترتيب النتائج.

SELECT last_name, salary
FROM employees
WHERE salary > 10000
ORDER BY salary DESC;
